# 外卖 Agent 评估实验报告

## 1. 实验概述

本实验针对一个外卖 Agent（Food Delivery Agent）场景，基于课程提供的 27 个评估样本，自行实现了 3 种评估指标对 Agent 的运行轨迹进行量化评估。评估指标参考了开源项目 Ragas 的设计思路，但未直接调用其顶层 API，均为自主实现。

**评估环境：** LLM 后端使用本地 Claude CLI（claude-internal），通过 subprocess 调用。

---

## 2. 指标选取与实现

### 2.1 指标一：工具调用准确度（Tool Call Accuracy）

**选取原因：** 工具调用是 Agent 执行任务的核心动作。在外卖场景中，Agent 需要按正确的顺序调用 `search_restaurants`、`search_products`、`place_order`、`pay_order` 等工具，且传入正确的参数。该指标直接衡量 Agent "做对了什么"。

**实现过程（参考 Ragas `ToolCallAccuracy`）：**

该指标为纯规则指标，不依赖 LLM，包含两个子指标：

1. **Tool Call Accuracy（顺序匹配）：** 将实际执行的 `steps` 与期望的 `expected_steps` 按位置一一配对。对每一步计算：
   - 工具名称是否一致（精确匹配，权重 0.4）
   - 参数是否匹配（解析 input JSON 后逐字段归一化比较，权重 0.6）
   - 若步骤数不一致，多余/缺失的步骤记 0 分
   - 最终 Accuracy = 所有步骤得分的平均值

2. **Tool Call F1（无序匹配）：** 不考虑步骤顺序，将每步的工具名+参数生成签名，按集合方式计算 Precision、Recall、F1。这是一个更宽松的指标，适用于评估 Agent 是否"做了该做的事"而不拘泥于顺序。

### 2.2 指标二：规划合理性（Planning Rationality）

**选取原因：** 在多步任务中，Agent 不仅要调用正确的工具，还需要有合理的规划——不重复、不遗漏、逻辑连贯。该指标关注的是执行轨迹的"质量"而非仅仅是"正确性"。

**实现过程（自定义指标）：**

该指标为纯规则+启发式指标，包含三个子指标：

1. **冗余检测（Redundancy Score）：** 检查是否存在连续重复的工具调用（相同的 tool_call + 相同的 input）。冗余率 = 冗余步骤数 / 总步骤数，得分 = 1.0 - 冗余率。

2. **步骤完整性（Completeness Score）：** 以 `expected_steps` 的工具调用序列为参考，检查实际轨迹是否覆盖了所有必要步骤。采用多集合匹配（处理重复调用），覆盖率 = 已覆盖步骤数 / 期望步骤总数。

3. **逻辑连贯性（Coherence Score）：** 预定义工具间的依赖关系（如 `pay_order` 必须先有 `place_order`），检查两方面：
   - 工具级依赖：前置工具是否已执行
   - 参数级依赖：关键参数值是否出现在前序步骤的 observation 中（即参数是否有"来源"）

   连贯性得分 = 满足依赖的步骤数 / 有依赖要求的步骤总数。

4. **综合得分 = 0.3 × 冗余分 + 0.3 × 完整性分 + 0.4 × 连贯性分**（连贯性权重较高，因为逻辑断层是最严重的问题）。

### 2.3 指标三：任务完成度（Task Completion）

**选取原因：** 最终用户关心的是"任务到底完成了没有"。即使工具调用全对、规划也合理，如果最终回答遗漏了关键信息或误解了用户意图，任务仍然是失败的。该指标从用户视角进行端到端评估。

**实现过程（参考 Ragas `AgentGoalAccuracyWithReference`）：**

该指标采用 LLM-as-Judge 方式：

1. 构造评估 Prompt，包含用户请求（`user_query`）、期望结果（`ground_truth`）、Agent 最终回答（`final_answer`）、执行步骤摘要
2. 要求 LLM 综合判断任务是否完成，输出结构化 JSON `{"verdict": 0/1, "reason": "..."}`
3. 对全数据集汇总计算完成率

## 3. 评估结果

### 3.1 总体结果

| 指标 | 平均分/完成率 |
|------|-------------|
| 工具调用准确度（Accuracy） | **0.6875** |
| 工具调用 F1 | **0.6987** |
| 规划合理性（综合） | **0.8571** |
| 任务完成度 | **57.69%**（15/26 有效样本） |

### 3.2 工具调用准确度详细分析

- **完美匹配（Accuracy = 1.0）：** 8/27 个样本，主要是简单的单步查询任务（如查订单状态、搜索餐厅）
- **零分样本（Accuracy = 0.0）：** 2 个样本（c3d4e5f6、d4e5f6a7），均为实际步骤与期望步骤严重偏离的情况
- **Accuracy vs F1 的差异：** F1（0.6987）略高于 Accuracy（0.6875），说明部分样本的工具调用本身是对的，只是顺序或数量与期望不一致。例如样本 c3d4e5f6 的 Accuracy 为 0（顺序完全不对），但 F1 达到 0.89（实际调用的工具大部分是正确的）

**典型问题样本：**
- `7efa3d78`：Accuracy 仅 0.14，但 F1 为 0.75 —— Agent 做了正确的工具调用但顺序混乱
- `d377268a`：Accuracy 0.25 —— Agent 跳过了 `search_products` 直接下单

### 3.3 规划合理性详细分析

- **冗余检测：** 平均得分 1.0（满分），即数据集中没有出现连续重复调用的情况，Agent 没有陷入死循环
- **步骤完整性：** 平均 0.935，大多数样本覆盖了期望的工具类型。缺失主要出现在多步任务中遗漏了 `pay_order` 步骤
- **逻辑连贯性：** 平均 0.691，是三个子指标中最低的。**24/27 个样本存在违规**，主要分两类：
  1. **前置依赖缺失（missing_prerequisite）：** 8 个样本直接调用了 `pay_order` 或 `check_order_status` 而未先执行 `place_order`，这些是用户直接要求"支付订单XXX"或"查询订单XXX"的场景——Agent 直接操作了，但从轨迹看它没有上游步骤
  2. **参数无来源（param_not_grounded）：** 24 个样本存在 `items` 参数（如 `{'P201': 1}`）无法从前序 observation 中找到的情况。这是因为 `place_order` 的 items 参数中的 product_id 虽然出现在 `search_products` 的返回结果中，但以字典格式传入时字符串匹配无法捕捉到

> **说明：** `param_not_grounded` 的高发率部分源于检测逻辑较为严格——它用简单的字符串包含判断参数值是否出现在 observation 中，而 `items` 字段是 dict 类型（如 `{'P201': 1}`），其字符串表示与 observation 中 `"product_id": "P201"` 的格式不同。这属于检测方法的局限性而非 Agent 真正的错误。

### 3.4 任务完成度详细分析

- **完成率 57.69%**（15 完成 / 11 未完成 / 1 评估失败）
- 评估失败的 1 个样本（`a1a3d14a`）是因为 LLM 未返回 JSON 格式输出

**LLM 判定为"未完成"的典型原因：**

| 原因类别 | 涉及样本数 | 示例 |
|---------|-----------|------|
| 缺少支付步骤 | 4 | 用户要求微信/支付宝支付但 Agent 未执行 |
| 逻辑错误 | 2 | 跳过必要步骤直接操作（如未查菜品就下单） |
| 违背用户意图 | 2 | 用户说"暂不支付"但 Agent 催促支付 |
| 信息不完整 | 2 | 最终回答缺少价格、餐厅名等关键信息 |
| 部分需求遗漏 | 1 | 用户问了两个问题只回答了一个 |

## 4. 综合分析与发现

### 4.1 三个指标间的关联

通过对比各样本在三个指标上的表现，可以观察到：

1. **工具调用准确但任务未完成的情况确实存在。** 例如样本 `42966cef`，Accuracy 0.64（中等），规划合理性 0.93（较高），但 LLM 判定任务未完成——原因是缺少支付步骤。这说明 Accuracy 是必要条件但非充分条件。

2. **规划合理性高不等于任务完成。** 样本 `04000009` 和 `378e19cf` 的完整性和冗余分都是满分，但连贯性为 0（直接调用末端工具），LLM 也判定为未完成。

3. **简单任务（单步）的三个指标往往一致。** 查询类任务（查订单状态、搜餐厅）通常三个指标都表现良好。

### 4.2 Agent 的主要不足

1. **多步任务的步骤遗漏**是最突出的问题——尤其是支付步骤经常被跳过
2. **对用户意图的细粒度理解不足**——如忽略"暂不支付"的限定、只完成部分请求
3. **工具调用顺序**在复杂任务中有时会混乱

### 4.3 评估方法的局限性

1. **参数匹配过于严格：** 连贯性检测中的字符串匹配无法处理格式差异（dict vs JSON 字符串），导致大量误报
2. **LLM 评估的稳定性：** 1/27 样本出现 JSON 提取失败，且 LLM 的判断标准可能因 prompt 措辞而波动
3. **数据集规模有限：** 27 个样本难以得出统计显著的结论

---

## 5. 改进方向

1. **连贯性检测**可引入更智能的参数匹配——解析 JSON 结构后按字段匹配而非纯字符串包含
2. **任务完成度**可增加多维度评分（不只是 0/1），例如区分"核心任务完成但细节不足"和"完全失败"
3. **增加更多指标**如思维链质量（thought 与 action 的一致性）、执行效率（实际步骤数 vs 最优步骤数）
